What is Database Management System (DBMS)?

A Database Management System (DBMS) is software that enables users to store, retrieve, and manage data efficiently. It provides a systematic way to create, retrieve, update and manage data in a database.

Key Components:

  • Database: Collection of related data
  • Database Engine: Core service for storing and retrieving data
  • Database Schema: Structure that defines the organization of data
  • Query Processor: Processes and executes database queries

Learn the core concepts and foundation of database systems

Introduction to DBMS

A Database Management System is a software application that interacts with users, applications, and the database itself to capture and analyze data.

Why do we need DBMS?

  • Data Organization: Structures data in a logical manner
  • Data Security: Provides access control and data protection
  • Data Integrity: Ensures data accuracy and consistency
  • Concurrent Access: Multiple users can access data simultaneously
  • Backup & Recovery: Protects against data loss
Real-world Example: Banking systems use DBMS to manage customer accounts, transactions, and maintain data consistency across all operations.

Database vs File System

Before databases, data was stored in file systems. However, file systems have several limitations that databases solve:

File System Problems:

  • Data Redundancy: Same data stored in multiple files leads to inconsistency
  • Data Isolation: Data scattered across files makes access difficult
  • Integrity Problems: Hard to enforce data validation rules
  • Atomicity Problems: Partial updates can leave data in inconsistent state
  • Concurrent Access Issues: Multiple users can corrupt data
  • Security Problems: Difficult to provide selective access
File System Example: student_personal.txt: ID=1, Name=John, Age=20 student_academic.txt: ID=1, Name=John, GPA=3.5, Course=CS student_financial.txt: ID=1, Name=Jhon, Fees=5000 Problem: Name inconsistency (John vs Jhon)

DBMS Solutions:

  • Centralized data storage eliminates redundancy
  • Standardized access methods through SQL
  • Built-in integrity constraints
  • Transaction management for atomicity
  • Concurrent access control
  • User authentication and authorization

DBMS Architecture

DBMS architecture describes how a database system is designed and organized. There are two main architectural approaches:

1. Three-Schema Architecture (ANSI-SPARC)

  • External Level (View Level): Individual user views of the database
    -- Teacher's view CREATE VIEW TeacherView AS SELECT student_name, course_name, grade FROM students s, courses c, enrollments e WHERE s.student_id = e.student_id AND c.course_id = e.course_id;
  • Conceptual Level (Logical Level): Complete logical structure of entire database
    -- Conceptual schema Students(student_id, name, email, department) Courses(course_id, course_name, credits) Enrollments(student_id, course_id, grade, semester)
  • Internal Level (Physical Level): How data is actually stored on storage devices
    -- Physical storage details Students table: stored as B-tree index on student_id Location: /data/students.dat Block size: 4KB, Records per block: 50

2. Three-Tier Architecture

  • Presentation Tier: User interface (web browser, mobile app)
  • Application Tier: Business logic and processing
  • Data Tier: Database server where data is stored
Data Independence: Changes at lower levels don't affect higher levels. For example, changing how data is physically stored doesn't affect application code.

Data Models

Data models define how data is structured, stored, and manipulated in a database system.

1. Hierarchical Model

  • Data organized in tree-like structure
  • Parent-child relationships (one-to-many)
  • Example: File systems, organizational charts
  • Limitation: Rigid structure, difficult to represent many-to-many relationships

2. Network Model

  • Extension of hierarchical model
  • Allows many-to-many relationships
  • Uses graph structure with nodes and edges
  • More flexible but complex to implement

3. Relational Model

  • Most popular model used today
  • Data stored in tables (relations) with rows and columns
  • Based on mathematical set theory
  • Examples: MySQL, PostgreSQL, Oracle, SQL Server
-- Relational Model Example Students Table: +------------+----------+-------------------+ | student_id | name | email | +------------+----------+-------------------+ | 1 | John Doe | john@example.com | | 2 | Jane | jane@example.com | +------------+----------+-------------------+

4. Object-Oriented Model

  • Combines object-oriented programming with database features
  • Supports complex data types, inheritance, and encapsulation
  • Examples: ObjectDB, db4o

5. NoSQL Models

  • Document: JSON/XML documents (MongoDB)
  • Key-Value: Simple key-value pairs (Redis)
  • Column-Family: Column-oriented storage (Cassandra)
  • Graph: Nodes and relationships (Neo4j)

Entity-Relationship (ER) Model

The ER model is a high-level conceptual data model that helps in designing databases by representing the real world as entities and relationships.

Core Components:

  • Entity: A thing or object in the real world that can be identified uniquely
    Examples: - Student (John, Mary, David) - Course (Database Systems, Data Structures) - Department (Computer Science, Mathematics)
  • Attributes: Properties that describe an entity
    Student attributes: - student_id (identifier) - name (simple) - address (composite: street, city, state) - phone_numbers (multi-valued) - age (derived from birth_date)
  • Relationships: Associations between entities
    Examples: - Student "enrolls in" Course - Professor "teaches" Course - Department "offers" Course

Attribute Types:

  • Simple: Cannot be divided further (name, age)
  • Composite: Can be divided into components (address = street + city + state)
  • Single-valued: Has one value (student_id)
  • Multi-valued: Can have multiple values (phone_numbers)
  • Derived: Calculated from other attributes (age from birth_date)

ER Diagrams

ER diagrams provide a visual representation of the database design using standardized symbols.

ER Diagram Symbols:

  • Rectangle: Entity sets
  • Ellipse: Attributes
  • Diamond: Relationship sets
  • Lines: Connect entities to attributes and relationships
  • Double Rectangle: Weak entity sets
  • Double Ellipse: Multi-valued attributes
  • Dashed Ellipse: Derived attributes
  • Underline: Primary key attributes
ER Diagram Example (text representation): [Student] ---- (enrolls) ---- [Course] | | | | student_id course_id name course_name email credits
Benefits of ER Diagrams: They provide clear communication between database designers and stakeholders, help identify data requirements, and serve as a blueprint for database implementation.

Entity Types: Strong vs Weak

Strong Entity:

  • Has its own primary key
  • Can exist independently
  • Represented by single rectangle
Strong Entity Example: Student - student_id (Primary Key) - name - email - department Can exist without depending on other entities

Weak Entity:

  • Cannot be uniquely identified by its own attributes
  • Depends on a strong entity (owner entity)
  • Uses partial key + owner's key for identification
  • Represented by double rectangle
Weak Entity Example: Dependent (employee's family member) - dependent_name (Partial Key) - relationship - age Depends on Employee entity: - employee_id + dependent_name = Unique identification
Important: Weak entities cannot exist without their owner entity. If an employee is deleted, all their dependents must also be deleted.

Relationships and Cardinality

Relationships define how entities are connected, and cardinality specifies how many instances of each entity can be associated.

Cardinality Types:

1. One-to-One (1:1)

  • Each entity instance is associated with exactly one instance of another entity
  • Example: Person ‚Üî Passport
One-to-One Example: Person(person_id, name, address) Passport(passport_no, issue_date, expiry_date, person_id) Each person has exactly one passport Each passport belongs to exactly one person

2. One-to-Many (1:M)

  • One entity instance can be associated with multiple instances of another entity
  • Example: Department ‚Üí Employees
One-to-Many Example: Department(dept_id, dept_name, location) Employee(emp_id, name, salary, dept_id) One department has many employees Each employee belongs to one department

3. Many-to-Many (M:N)

  • Multiple instances of each entity can be associated with multiple instances of the other
  • Example: Students ‚Üî Courses
  • Requires a junction table in relational implementation
Many-to-Many Example: Student(student_id, name, email) Course(course_id, course_name, credits) Enrollment(student_id, course_id, grade, semester) One student can enroll in many courses One course can have many students

Participation Constraints:

  • Total Participation: Every entity must participate (double line)
  • Partial Participation: Some entities may not participate (single line)

SQL Fundamentals

SQL (Structured Query Language) is a standardized programming language designed for managing and manipulating relational databases.

Key Characteristics:

  • Declarative: You specify what you want, not how to get it
  • Standardized: ANSI/ISO standard with vendor variations
  • Comprehensive: Handles data definition, manipulation, and control
  • Set-based: Operates on sets of data, not individual records

SQL Command Categories:

  • DDL (Data Definition Language): CREATE, ALTER, DROP
  • DML (Data Manipulation Language): SELECT, INSERT, UPDATE, DELETE
  • DCL (Data Control Language): GRANT, REVOKE
  • TCL (Transaction Control Language): COMMIT, ROLLBACK, SAVEPOINT
-- Basic SQL Query Structure SELECT column1, column2, ... FROM table_name WHERE condition GROUP BY column HAVING group_condition ORDER BY column;

Data Definition Language (DDL)

DDL commands are used to define and modify database structure.

CREATE - Creating Database Objects

-- Create Database CREATE DATABASE university; -- Create Table CREATE TABLE students ( student_id INT PRIMARY KEY, first_name VARCHAR(50) NOT NULL, last_name VARCHAR(50) NOT NULL, email VARCHAR(100) UNIQUE, birth_date DATE, gpa DECIMAL(3,2) CHECK (gpa >= 0.0 AND gpa <= 4.0), enrollment_date DATE DEFAULT CURRENT_DATE ); -- Create Index CREATE INDEX idx_student_email ON students(email); CREATE INDEX idx_student_name ON students(last_name, first_name);

ALTER - Modifying Database Objects

-- Add Column ALTER TABLE students ADD COLUMN phone VARCHAR(15); -- Modify Column ALTER TABLE students MODIFY COLUMN email VARCHAR(150); -- Drop Column ALTER TABLE students DROP COLUMN phone; -- Add Constraint ALTER TABLE students ADD CONSTRAINT chk_gpa CHECK (gpa BETWEEN 0.0 AND 4.0);

DROP - Removing Database Objects

-- Drop Table DROP TABLE students; -- Drop Index DROP INDEX idx_student_email; -- Drop Database DROP DATABASE university;

Data Manipulation Language (DML)

DML commands are used to manipulate data within database tables.

INSERT - Adding Data

-- Insert Single Record INSERT INTO students (student_id, first_name, last_name, email, gpa) VALUES (1, 'John', 'Doe', 'john.doe@email.com', 3.75); -- Insert Multiple Records INSERT INTO students (student_id, first_name, last_name, email, gpa) VALUES (2, 'Jane', 'Smith', 'jane.smith@email.com', 3.90), (3, 'Bob', 'Johnson', 'bob.johnson@email.com', 3.25), (4, 'Alice', 'Brown', 'alice.brown@email.com', 3.85); -- Insert from Another Table INSERT INTO honor_students (student_id, name, gpa) SELECT student_id, CONCAT(first_name, ' ', last_name), gpa FROM students WHERE gpa >= 3.5;

SELECT - Retrieving Data

-- Basic SELECT SELECT first_name, last_name, gpa FROM students; -- SELECT with Conditions SELECT * FROM students WHERE gpa > 3.5 AND enrollment_date >= '2023-01-01'; -- SELECT with Sorting SELECT first_name, last_name, gpa FROM students ORDER BY gpa DESC, last_name ASC; -- SELECT with LIMIT SELECT * FROM students ORDER BY gpa DESC LIMIT 5;

UPDATE - Modifying Data

-- Update Single Record UPDATE students SET gpa = 3.80, email = 'john.doe.updated@email.com' WHERE student_id = 1; -- Update Multiple Records UPDATE students SET gpa = gpa + 0.1 WHERE enrollment_date < '2023-01-01'; -- Update with Subquery UPDATE students SET gpa = (SELECT AVG(grade) FROM enrollments WHERE enrollments.student_id = students.student_id) WHERE student_id IN (SELECT student_id FROM enrollments);

DELETE - Removing Data

-- Delete Specific Records DELETE FROM students WHERE gpa < 2.0; -- Delete with Subquery DELETE FROM students WHERE student_id NOT IN (SELECT DISTINCT student_id FROM enrollments); -- Delete All Records (but keep table structure) DELETE FROM students;

SQL Functions and Advanced Queries

SQL provides various built-in functions for data manipulation and analysis.

Aggregate Functions

-- Basic Aggregate Functions SELECT COUNT(*) as total_students, AVG(gpa) as average_gpa, MAX(gpa) as highest_gpa, MIN(gpa) as lowest_gpa, SUM(credits) as total_credits FROM students; -- GROUP BY with Aggregates SELECT department, COUNT(*) as student_count, AVG(gpa) as avg_gpa, MAX(gpa) as max_gpa FROM students GROUP BY department HAVING COUNT(*) > 10;

String Functions

-- String Manipulation SELECT UPPER(first_name) as uppercase_name, LOWER(last_name) as lowercase_surname, LENGTH(email) as email_length, SUBSTRING(email, 1, POSITION('@' IN email) - 1) as username, CONCAT(first_name, ' ', last_name) as full_name, REPLACE(phone, '-', '.') as formatted_phone FROM students;

Date Functions

-- Date Operations SELECT first_name, birth_date, CURRENT_DATE as today, TIMESTAMPDIFF(YEAR, birth_date, CURRENT_DATE) as age, DATE_ADD(enrollment_date, INTERVAL 4 YEAR) as expected_graduation, DAYOFWEEK(birth_date) as birth_day_of_week, MONTHNAME(birth_date) as birth_month FROM students;

SQL Joins

Joins are used to combine data from multiple tables based on related columns.

INNER JOIN

Returns only the matching records from both tables.

-- INNER JOIN Example SELECT s.first_name, s.last_name, c.course_name, e.grade FROM students s INNER JOIN enrollments e ON s.student_id = e.student_id INNER JOIN courses c ON e.course_id = c.course_id;

LEFT JOIN (LEFT OUTER JOIN)

Returns all records from the left table and matching records from the right table.

-- LEFT JOIN Example SELECT s.first_name, s.last_name, c.course_name, e.grade FROM students s LEFT JOIN enrollments e ON s.student_id = e.student_id LEFT JOIN courses c ON e.course_id = c.course_id; -- This shows all students, even those not enrolled in any course

RIGHT JOIN and FULL OUTER JOIN

-- RIGHT JOIN SELECT s.first_name, c.course_name FROM students s RIGHT JOIN enrollments e ON s.student_id = e.student_id RIGHT JOIN courses c ON e.course_id = c.course_id; -- FULL OUTER JOIN (MySQL doesn't support, use UNION) SELECT s.first_name, c.course_name FROM students s LEFT JOIN enrollments e ON s.student_id = e.student_id LEFT JOIN courses c ON e.course_id = c.course_id UNION SELECT s.first_name, c.course_name FROM students s RIGHT JOIN enrollments e ON s.student_id = e.student_id RIGHT JOIN courses c ON e.course_id = c.course_id;
Join Performance Tip: Always ensure that join conditions use indexed columns for better performance, especially on large tables.

What is Normalization?

Database normalization is the process of organizing data in a database to reduce redundancy and improve data integrity. It involves decomposing tables to eliminate undesirable characteristics like insertion, update, and deletion anomalies.

Goals of Normalization:

  • Eliminate Data Redundancy: Reduce duplicate data storage
  • Ensure Data Consistency: Prevent update anomalies
  • Minimize Storage Space: Efficient use of storage
  • Maintain Data Integrity: Enforce business rules
Example of Problems without Normalization:
If student information is repeated in every course enrollment record, updating a student's address requires changing multiple records, leading to potential inconsistencies.

Functional Dependencies

A functional dependency A ‚Üí B means that if we know the value of A, we can uniquely determine the value of B.

Example Table: Student_Course +------------+----------+------------+------------+----------+ | student_id | name | course_id | course_name| grade | +------------+----------+------------+------------+----------+ | 1 | John | CS101 | Programming| A | | 1 | John | MATH201 | Calculus | B | | 2 | Jane | CS101 | Programming| A | +------------+----------+------------+------------+----------+ Functional Dependencies: student_id ‚Üí name (student ID determines name) course_id ‚Üí course_name (course ID determines course name) {student_id, course_id} ‚Üí grade (both together determine grade)

Types of Functional Dependencies:

  • Trivial FD: A ‚Üí B where B ‚äÜ A
  • Non-trivial FD: A ‚Üí B where B ‚äÑ A
  • Partial Dependency: Non-key attribute depends on part of composite key
  • Transitive Dependency: A ‚Üí B and B ‚Üí C, therefore A ‚Üí C

First Normal Form (1NF)

A table is in 1NF if all attributes contain only atomic (indivisible) values and there are no repeating groups.

Rules for 1NF:

  • Each column should contain atomic values
  • Each column should have a unique name
  • Order of rows and columns doesn't matter
  • No duplicate rows
-- Violates 1NF (multiple values in phone column) Student_Info: +------------+----------+-------------------+ | student_id | name | phone_numbers | +------------+----------+-------------------+ | 1 | John | 123-456, 789-012 | | 2 | Jane | 555-123 | +------------+----------+-------------------+ -- Satisfies 1NF (atomic values) Students: +------------+----------+ | student_id | name | +------------+----------+ | 1 | John | | 2 | Jane | +------------+----------+ Student_Phones: +------------+-------------+ | student_id | phone | +------------+-------------+ | 1 | 123-456 | | 1 | 789-012 | | 2 | 555-123 | +------------+-------------+

Second Normal Form (2NF)

A table is in 2NF if it's in 1NF and every non-key attribute is fully functionally dependent on the entire primary key (no partial dependencies).

-- Violates 2NF (partial dependencies exist) Student_Course: +------------+----------+------------+------------+-------+ | student_id | name | course_id | course_name| grade | +------------+----------+------------+------------+-------+ | 1 | John | CS101 | Programming| A | | 1 | John | MATH201 | Calculus | B | | 2 | Jane | CS101 | Programming| A | +------------+----------+------------+------------+-------+ Primary Key: {student_id, course_id} Partial Dependencies: - student_id ‚Üí name (name depends only on student_id) - course_id ‚Üí course_name (course_name depends only on course_id) -- Satisfies 2NF (decomposed into separate tables) Students: +------------+----------+ | student_id | name | +------------+----------+ | 1 | John | | 2 | Jane | +------------+----------+ Courses: +------------+------------+ | course_id | course_name| +------------+------------+ | CS101 | Programming| | MATH201 | Calculus | +------------+------------+ Enrollments: +------------+------------+-------+ | student_id | course_id | grade | +------------+------------+-------+ | 1 | CS101 | A | | 1 | MATH201 | B | | 2 | CS101 | A | +------------+------------+-------+

Third Normal Form (3NF)

A table is in 3NF if it's in 2NF and no non-key attribute is transitively dependent on the primary key.

-- Violates 3NF (transitive dependency exists) Students: +------------+----------+----------+----------+----------+ | student_id | name | dept_code| dept_name| dept_head| +------------+----------+----------+----------+----------+ | 1 | John | CS | Comp Sci | Dr. Smith| | 2 | Jane | CS | Comp Sci | Dr. Smith| | 3 | Bob | MATH | Math | Dr. Jones| +------------+----------+----------+----------+----------+ Transitive Dependencies: student_id ‚Üí dept_code ‚Üí dept_name student_id ‚Üí dept_code ‚Üí dept_head -- Satisfies 3NF (eliminated transitive dependencies) Students: +------------+----------+----------+ | student_id | name | dept_code| +------------+----------+----------+ | 1 | John | CS | | 2 | Jane | CS | | 3 | Bob | MATH | +------------+----------+----------+ Departments: +----------+------------+----------+ | dept_code| dept_name | dept_head| +----------+------------+----------+ | CS | Comp Sci | Dr. Smith| | MATH | Math | Dr. Jones| +----------+------------+----------+
Benefits of 3NF: Eliminates most redundancy, prevents update anomalies, and is generally sufficient for most business applications.

What are Transactions?

A transaction is a logical unit of work that consists of one or more database operations that should be executed as a single, indivisible unit. Transactions ensure data consistency and integrity in database systems.

-- Example: Bank Transfer Transaction BEGIN TRANSACTION; -- Deduct money from sender's account UPDATE accounts SET balance = balance - 1000 WHERE account_id = 'A001'; -- Add money to receiver's account UPDATE accounts SET balance = balance + 1000 WHERE account_id = 'A002'; -- If both operations succeed COMMIT; -- If any operation fails -- ROLLBACK;

Transaction States:

  • Active: Transaction is executing
  • Partially Committed: After final statement executed
  • Committed: After successful completion
  • Failed: When transaction cannot proceed
  • Aborted: After rollback and database restored
  • Terminated: Final state after commit or abort

ACID Properties

ACID properties ensure that database transactions are processed reliably.

Atomicity

  • Transaction is treated as a single, indivisible unit
  • Either all operations complete successfully or none do
  • No partial updates are allowed
-- Atomicity Example BEGIN TRANSACTION; INSERT INTO orders (order_id, customer_id, total) VALUES (1001, 'C001', 500); INSERT INTO order_items (order_id, product_id, quantity) VALUES (1001, 'P001', 2); UPDATE inventory SET quantity = quantity - 2 WHERE product_id = 'P001'; COMMIT; -- If any statement fails, entire transaction is rolled back

Consistency

  • Database moves from one consistent state to another
  • All integrity constraints are maintained
  • Business rules are preserved

Isolation

  • Concurrent transactions don't interfere with each other
  • Each transaction appears to execute in isolation
  • Prevents interference between concurrent transactions

Durability

  • Once committed, changes are permanent
  • Data survives system failures
  • Changes are written to non-volatile storage

Concurrency Control Problems

When multiple transactions execute concurrently, several problems can occur without proper control mechanisms.

1. Lost Update Problem

-- Two transactions updating the same account Transaction T1: Transaction T2: READ balance (1000) READ balance (1000) balance = balance + 100 balance = balance - 50 WRITE balance (1100) WRITE balance (950) Result: T1's update is lost, final balance is 950 instead of 1050

2. Dirty Read Problem

-- T2 reads uncommitted data from T1 Transaction T1: Transaction T2: BEGIN BEGIN UPDATE balance = 500 READ balance (500) -- Dirty read ROLLBACK -- T2 read data that was rolled back -- T2 continues with incorrect data

3. Non-repeatable Read

-- T1 reads same data twice, gets different values Transaction T1: Transaction T2: BEGIN BEGIN READ balance (1000) UPDATE balance = 500 COMMIT READ balance (500) -- Different value in same transaction -- Same data, different values!

4. Phantom Read

-- T1 sees new rows that weren't there before Transaction T1: Transaction T2: BEGIN BEGIN SELECT COUNT(*) FROM accounts WHERE balance > 100 -- Returns 5 rows INSERT INTO accounts VALUES (..., 200) COMMIT SELECT COUNT(*) FROM accounts WHERE balance > 100 -- Returns 6 rows (phantom row appeared)

Locking and Deadlocks

Locking is a concurrency control mechanism that prevents conflicts between concurrent transactions.

Lock Types:

  • Shared Lock (S-lock): Multiple transactions can read the same data
  • Exclusive Lock (X-lock): Only one transaction can modify data
-- Lock Compatibility Matrix | S | X | ----|----|---| S | ‚úì | ‚úó | X | ‚úó | ‚úó | -- Example SELECT * FROM accounts WHERE id = 1; -- Acquires S-lock UPDATE accounts SET balance = 1000 WHERE id = 1; -- Acquires X-lock

Deadlock Example:

-- Two transactions waiting for each other Transaction T1: Transaction T2: LOCK TABLE A LOCK TABLE B -- waiting for B -- waiting for A LOCK TABLE B LOCK TABLE A Result: Both transactions wait indefinitely (deadlock)

Deadlock Resolution:

  • Detection: Use wait-for graphs to detect cycles
  • Prevention: Order resources, use timeouts
  • Recovery: Abort one transaction (victim selection)

Database Indexing

Indexes are data structures that improve the speed of data retrieval operations on database tables.

How Indexes Work:

  • Create a separate structure pointing to table rows
  • Like an index in a book - helps find information quickly
  • Trade-off: Faster reads vs slower writes and more storage
-- Creating Indexes CREATE INDEX idx_student_lastname ON students(last_name); CREATE INDEX idx_student_email ON students(email); CREATE INDEX idx_composite ON students(department, gpa); -- Query that benefits from index SELECT * FROM students WHERE last_name = 'Smith'; -- Without index: Full table scan O(n) -- With index: Index lookup O(log n)

Types of Indexes:

  • B-Tree Index: Most common, good for range queries
  • Hash Index: Fast equality lookups, no range queries
  • Bitmap Index: Efficient for low-cardinality data
  • Composite Index: Multiple columns in single index
Index Best Practices: Create indexes on frequently queried columns, foreign keys, and columns used in WHERE clauses and JOIN conditions.

NoSQL Databases

NoSQL databases are designed to handle large volumes of unstructured or semi-structured data and provide horizontal scalability.

1. Document Databases

  • Store data as documents (JSON, BSON, XML)
  • Examples: MongoDB, CouchDB, Amazon DocumentDB
  • Use cases: Content management, catalogs, user profiles
// MongoDB Document Example { "_id": ObjectId("507f1f77bcf86cd799439011"), "name": "John Doe", "email": "john@example.com", "age": 25, "address": { "street": "123 Main St", "city": "Boston", "state": "MA" }, "courses": ["CS101", "MATH201", "PHYS301"], "enrollment_date": ISODate("2023-09-01") }

2. Key-Value Stores

  • Simple key-value pairs
  • Examples: Redis, Amazon DynamoDB, Riak
  • Use cases: Caching, session storage, real-time recommendations
// Redis Key-Value Examples SET user:1001:name "John Doe" SET user:1001:email "john@example.com" SET session:abc123 "user_id:1001,login_time:2023-09-01" // Retrieval GET user:1001:name // Returns "John Doe"

3. Column-Family Databases

  • Data organized by column families
  • Examples: Cassandra, HBase, Amazon SimpleDB
  • Use cases: Time-series data, IoT applications, large-scale analytics

4. Graph Databases

  • Store data as nodes and relationships
  • Examples: Neo4j, Amazon Neptune, ArangoDB
  • Use cases: Social networks, recommendation engines, fraud detection
// Neo4j Cypher Query Example // Create nodes and relationships CREATE (john:Person {name: 'John', age: 25}) CREATE (jane:Person {name: 'Jane', age: 23}) CREATE (john)-[:FRIENDS_WITH]->(jane) // Query relationships MATCH (p1:Person)-[:FRIENDS_WITH]->(p2:Person) RETURN p1.name, p2.name

Distributed Databases

Distributed databases store data across multiple physical locations, providing scalability and fault tolerance.

Distribution Strategies:

  • Horizontal Partitioning (Sharding): Split table rows across nodes
  • Vertical Partitioning: Split table columns across nodes
  • Replication: Copy data to multiple nodes
-- Horizontal Partitioning Example -- Partition users by region Users_North_America: user_id 1-100000 Users_Europe: user_id 100001-200000 Users_Asia: user_id 200001-300000 -- Queries are routed to appropriate partition SELECT * FROM users WHERE user_id = 50000; -- Routed to North America partition

CAP Theorem:

  • Consistency: All nodes see the same data simultaneously
  • Availability: System remains operational
  • Partition Tolerance: System continues despite network failures
  • Rule: Can only guarantee 2 out of 3 properties
CAP Theorem Trade-offs:
‚Ä¢ CA Systems: Traditional RDBMS (sacrifice partition tolerance)
‚Ä¢ CP Systems: MongoDB, HBase (sacrifice availability)
‚Ä¢ AP Systems: Cassandra, DynamoDB (sacrifice consistency)
